A URL shortener takes a long URL and returns a short alias that redirects to it.
It looks trivial, but it's a great warm-up problem because it forces you to reason
about read/write ratios, unique key generation, and caching — all under a tight
latency budget.

## Requirements

**Functional**

- Create a short URL for a given long URL.
- Redirect a short URL to its original long URL.
- Optionally support custom aliases and expiration.

**Non-functional**

- Redirects must be fast (low single-digit ms) and highly available.
- The system is **read-heavy** — far more redirects than creations.

## Back-of-the-envelope

Assume 100M new URLs/month and a 100:1 read:write ratio.

| Metric            | Estimate                              |
| ----------------- | ------------------------------------- |
| Writes            | 100M / month ≈ **40 writes/sec**      |
| Reads (redirects) | 100:1 ≈ **4,000 reads/sec**           |
| Storage (5 yrs)   | 100M × 60 × ~500 bytes ≈ **3 TB**     |

The headline: this is a read-heavy, cache-friendly workload. Optimize the read path.

The redirect path hits Redis first; only cache misses touch the database. With a
hot key set that fits in memory, the vast majority of reads never reach the store.

## Key generation

The interesting part is generating a short, unique, collision-free key.

1. **Hash + truncate** — hash the long URL (e.g. MD5) and take the first 7
   base62 characters. Simple, but collisions need a retry loop.
2. **Counter + base62** — maintain a distributed counter and base62-encode it.
   No collisions, but the IDs are sequential and guessable.
3. **Pre-generated keys (KGS)** — a Key Generation Service produces random unique
   keys offline and hands them out. Removes collision checks from the write path.

Base62 over 7 characters gives 62⁷ ≈ **3.5 trillion** keys — plenty.

```ts
const ALPHABET =
  "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ"

function encode(id: number): string {
  let s = ""
  while (id > 0) {
    s = ALPHABET[id % 62] + s
    id = Math.floor(id / 62)
  }
  return s.padStart(7, "0")
}
```

## The read path

For redirects, prefer a **301 vs 302** decision deliberately:

- `301 Moved Permanently` lets browsers cache the redirect — fewer hits on your
  servers, but you lose per-click analytics.
- `302 Found` routes every click through you — better analytics, more load.

> Most shorteners choose `302` so they can count clicks. If analytics don't
> matter, `301` is cheaper.

## Where it can go wrong

- **Hot keys** — a viral link can hammer one cache shard. Mitigate with
  replication or client-side caching of the redirect.
- **Counter as a single point of failure** — shard the counter or use ranges
  handed out to each app server.

That's the core. From here you can extend into analytics pipelines, rate
limiting, and custom-domain support.
